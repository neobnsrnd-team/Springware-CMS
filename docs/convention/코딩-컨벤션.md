## 1. 언어 & 주석

### 기본 원칙
- **주석은 한국어**로 작성합니다.
- 라이브러리·프레임워크 API 이름, 기술 용어는 영문 그대로 유지합니다.
- JSDoc(`/**`) 형식은 repository 함수 등 공개 함수에만 사용합니다. 내부 헬퍼는 일반 `//` 주석으로 충분합니다.

```ts
// ✅ 좋은 예
/** 페이지 단건 조회 */
export async function getPageById(pageId: string): Promise<CmsPage | null> { ... }

// 보안: 디렉토리 트래버설 방지
if (path.includes('..')) { ... }

// ❌ 나쁜 예 (영어 주석)
// Security: prevent directory traversal
if (path.includes('..')) { ... }
```

### 파일 상단 경로 주석
각 파일 첫 줄에 해당 파일의 경로를 주석으로 명시합니다. 파일을 열지 않고도 위치를 파악할 수 있습니다.

```ts
// src/app/api/builder/save/route.ts
```

```
1. 외부 라이브러리 (react, next/*, oracledb, axios 등)
```

```ts
import { NextRequest } from 'next/server';
import { getConnection } from '@/db/connection';
```

## 5. TypeScript 타입 작성

### interface vs type
- **`interface`** 우선 사용: 컴포넌트 Props, API 요청/응답 body, DB 모델
- **`type`**: 유니온 타입, 복잡한 조합형에만 사용

```ts
// ✅ Props는 interface
export interface FileBrowserProps {
    apiEndpoints?: Partial<ApiEndpoints>;
    className?: string;
}

// ✅ 유니온은 type
type ApproveState = 'DRAFT' | 'PENDING' | 'APPROVED' | 'REJECTED';
```

### 타입 선언 위치
- 파일 내부에서만 쓰는 타입은 해당 파일 상단에 선언합니다.
- 여러 파일에서 쓰는 타입은 `src/db/types.ts` (DB 타입) 또는 해당 도메인 파일에 정의 후 export합니다.

### catch 블록 에러 처리
- catch 변수명은 **`err: unknown`으로 통일**합니다. (`error` 사용 금지, 타입 명시 필수)
- 에러 메시지 추출은 반드시 `getErrorMessage(err)` 헬퍼를 사용합니다. (`instanceof Error` 직접 사용 금지)

```ts
// ✅ 좋은 예
import { getErrorMessage } from '@/lib/api-response';

} catch (err: unknown) {
    console.error('실패:', err);
    return NextResponse.json({ ok: false, error: getErrorMessage(err) }, { status: 500 });
}

// ❌ 나쁜 예 (변수명 error 금지, instanceof 직접 사용 금지, any 금지)
} catch (error) { ... }
} catch (error: any) { ... }
return NextResponse.json({ error: error instanceof Error ? error.message : '알 수 없는 오류' });
```

### 옵셔널 chaining & nullish coalescing
- `?.` 와 `??` 를 적극 활용합니다.

```ts
// ✅
const html = (page.FILE_PATH ? await readPageHtml(page.FILE_PATH) : null) ?? page.PAGE_DESC ?? '';
const bank = isValidBankId(body.bank) ? body.bank : 'ibk';
```

## 6. API Route 구조

### 파일 구조 템플릿

```ts
// src/app/api/{도메인}/{액션}/route.ts

import { NextRequest } from 'next/server';
import { successResponse, errorResponse, getErrorMessage } from '@/lib/api-response';

// 모듈 스코프 상수 (환경변수, 설정값)
const SOME_CONSTANT = process.env.SOME_ENV || '기본값';

// 입력 검증 함수 (필요한 경우)
function isValidInput(value: unknown): value is string {
    return typeof value === 'string' && /^[a-z0-9-]{1,64}$/.test(value);
}

// 비즈니스 로직 함수 (핸들러에서 분리)
async function doSomething(input: string): Promise<결과타입> {
    // ...
}

// HTTP 메서드 핸들러
export async function POST(req: NextRequest) {
    try {
        const body = await req.json();
        // ...
        return successResponse();
    } catch (err: unknown) {
        console.error('작업 실패:', err);
        return errorResponse(getErrorMessage(err));
    }
}
```

### 핵심 원칙
- **핸들러와 비즈니스 로직을 분리합니다.** 핸들러(`POST`, `GET` 등)는 요청 파싱·응답 반환만 담당하고, 실제 처리는 별도 함수로 분리합니다.
- **같은 도메인 리소스의 메서드는 한 파일에 둘 수 있습니다.** 예) `pages/route.ts`에 GET(목록)과 DELETE(삭제)를 함께 정의.
- **입력 검증은 핸들러 진입부에서 먼저 수행합니다.**

## 7. 응답 형식

### 성공 응답 — `successResponse` 헬퍼 사용

`src/lib/api-response.ts`의 `successResponse` 헬퍼를 사용합니다. `ok: true`가 자동으로 포함되며, `success: true`와 혼용하지 않습니다.

```ts
import { successResponse } from '@/lib/api-response';

// ✅ 통일된 성공 응답
return successResponse();                           // { ok: true }
return successResponse({ html, updated });          // { ok: true, html, updated }
return successResponse({ deleted: count });         // { ok: true, deleted: count }

// ❌ 사용 금지 (직접 NextResponse.json 또는 success 필드 사용)
return NextResponse.json({ ok: true });
return NextResponse.json({ success: true });
```

### 에러 응답 — 두 가지 패턴 구분

| 상황 | 헬퍼 함수 | 형식 | HTTP 상태코드 |
|------|----------|------|-------------|
| ContentBuilder 라이브러리에 직접 연결된 API | `contentBuilderErrorResponse(msg)` | `{ ok: false, error: "메시지" }` | **200** |
| 일반 Next.js API | `errorResponse(msg, status)` | `{ ok: false, error: "메시지" }` | 400 / 500 |

```ts
import { errorResponse, contentBuilderErrorResponse } from '@/lib/api-response';

// ContentBuilder 규약 API (라이브러리가 body.ok로 판별, HTTP 200 필수)
return contentBuilderErrorResponse('업로드 실패');

// 일반 API — 헬퍼 사용 (NextResponse.json 직접 사용 금지)
return errorResponse('잘못된 요청입니다.', 400);
return errorResponse('서버 오류가 발생했습니다.');  // 기본 status: 500
```

### 부분 성공 응답 (배치 작업)

```ts
// 일부 성공, 일부 실패인 경우 — successResponse 헬퍼 사용
return successResponse({
    deleted: deletedCount,
    failed: failedCount,
    errors,           // 실패한 항목 상세
});
// → { ok: true, deleted: N, failed: M, errors: [...] }
```

### 에러 메시지 작성 규칙
- **한국어**로 작성합니다.
- 민감한 정보(스택 트레이스, DB 쿼리, 파일 경로 등)를 노출하지 않습니다.
- 사용자가 조치할 수 있는 내용을 담습니다.

```ts
// ✅ 좋은 예
{ error: '파일 크기가 초과되었습니다. 최대 100MB까지 업로드 가능합니다.' }
{ error: '유효하지 않은 경로입니다.' }

// ❌ 나쁜 예 (내부 정보 노출)
{ error: `ENOENT: no such file or directory, open '${filePath}'` }
```

## 8. 에러 처리

### 기본 패턴

```ts
import { errorResponse, getErrorMessage } from '@/lib/api-response';

try {
    // ...
} catch (err: unknown) {
    console.error('한국어로 설명:', err); // 서버 로그에는 상세 기록
    return errorResponse(getErrorMessage(err));
}
```

### console.error 사용 기준
- `console.error`: 예상치 못한 서버 오류 (try/catch catch 블록)
- `console.warn`: 예상 가능하지만 주의가 필요한 상황 (예: 파일 없음 → 기본값 사용)
- `console.log`: 개발 중 디버깅 전용. **커밋 전 반드시 제거**

## 9. DB 레이어 패턴

### 구조 원칙
- **SQL 쿼리**는 `src/db/queries/*.sql.ts`에 상수로 정의합니다. repository 안에 SQL 문자열을 직접 쓰지 않습니다.
- **데이터 접근**은 `src/db/repository/*.repository.ts`에서만 수행합니다. API route에서 직접 DB 커넥션을 사용하지 않습니다.

```
API route → repository → queries (SQL 상수) → DB
```

### 커넥션 관리

```ts
// ✅ 단순 조회 — getConnection() + finally conn.close()
export async function getPageById(pageId: string): Promise<CmsPage | null> {
    const conn = await getConnection();
    try {
        const result = await conn.execute<CmsPage>(PAGE_SELECT_BY_ID, { pageId }, OBJ);
        return result.rows?.[0] ?? null;
    } finally {
        await conn.close(); // finally에서 반드시 close
    }
}

// ✅ 쓰기 작업 — withTransaction() 콜백 패턴
export async function createPage(input: { ... }): Promise<void> {
    await withTransaction(async (conn) => {
        await conn.execute(PAGE_INSERT, { ... });
        await conn.execute(PAGE_HISTORY_INSERT, { ... }); // 한 트랜잭션으로 묶음
    });
}
```

### CLOB 처리
- CLOB 컬럼 바인딩은 반드시 `clobBind()` 래퍼를 사용합니다.

```ts
await conn.execute(PAGE_UPDATE, {
    pageDesc: clobBind(input.pageDesc ?? null),
    rejectedReason: clobBind(input.rejectedReason ?? null),
});
```

### 결과 형식
- `OUT_FORMAT_OBJECT`는 파일 상단에 `const OBJ = { outFormat: oracledb.OUT_FORMAT_OBJECT }` 상수로 정의합니다.
- DB 컬럼명은 Oracle 기본 대문자(`PAGE_ID`, `FILE_PATH`)를 그대로 사용합니다.

## 10. React 컴포넌트 패턴

### 파일 구조 템플릿

```tsx

interface InternalType {
    name: string;

export interface MyComponentProps {
    value: string;
    onChange?: (value: string) => void;
}

export default function MyComponent({ value, onChange }: MyComponentProps) {
    const [localState, setLocalState] = useState('');

    const handleChange = useCallback(() => {
        // ...
    }, []);

            {/* JSX */}
        </div>

```

### 핵심 원칙
- **서버 컴포넌트가 기본.** 브라우저 API·이벤트 핸들러·state가 필요한 경우에만 `'use client'` 선언합니다.
- **SSR 불가 라이브러리**(ContentBuilder 등)는 `dynamic(..., { ssr: false })` 래퍼 컴포넌트를 별도로 만듭니다.
- **Props 타입은 반드시 interface로 export.** 다른 파일에서 참조할 때 편리합니다.
- **default export**로 컴포넌트를 반환합니다.

### hooks 사용 순서
```tsx
// 선언 순서를 통일합니다
const [상태] = useState(...)       // 1. state
const 참조 = useRef(...)           // 2. ref
const 콜백 = useCallback(...)      // 3. callback
const 계산값 = useMemo(...)        // 4. memo
useEffect(...)                     // 5. effect (마지막)
```

## 11. 보안 체크리스트

파일 경로나 외부 입력을 다루는 코드 작성 시 반드시 확인합니다.

### 디렉토리 트래버설 방지

```ts
import { errorResponse } from '@/lib/api-response';

// 경로 파라미터에 반드시 포함
if (relativePath.includes('..')) {
    return errorResponse('유효하지 않은 경로입니다.', 400);
}

// 절대 경로 확인 (더 엄격한 경우 — path.sep으로 접두사 매칭 취약점 방지)
const baseDir = path.resolve(ASSET_UPLOAD_DIR) + path.sep;
const absolutePath = path.resolve(baseDir, relativePath);
if (!absolutePath.startsWith(baseDir)) {
    return errorResponse('유효하지 않은 경로입니다.', 400);
}
```

### ID 검증 정규식
```ts
// bank ID, page ID 등 경로에 사용되는 식별자 검증
function isValidId(id: unknown): id is string {
    return typeof id === 'string' && /^[a-z0-9-]{1,64}$/.test(id);
}
```

### 파일명 새니타이징
```ts
// 업로드 파일명 정규화 (특수문자 → _)
const filename = file.name.replace(/[^a-zA-Z0-9._-]/g, '_');
```

### API 키
- 모든 API 키는 서버 `.env`에만 저장합니다.
- 클라이언트 컴포넌트에서 `process.env.XXX_API_KEY`를 직접 참조하지 않습니다.
- `NEXT_PUBLIC_` prefix가 없는 환경변수는 서버 전용입니다.

## 13. E2E 테스트 코드 (Playwright)

> 상세 아키텍처 및 구조적 한계는 [`docs/reference/E2E-테스트-아키텍처.md`](../reference/E2E-테스트-아키텍처.md) 참조.

### 파일 구조

```
e2e/
├── api/          # API 계약 테스트 (page.route() mock)
├── pages/        # 페이지·플로우 테스트
├── smoke/        # 스모크 테스트 (서버 기동 확인)
└── fixtures/
    └── locale.ts # UI 텍스트 중앙 관리 (셀렉터 문자열)
```

### mock 데이터 작성 기준

`page.route()` mock 응답은 **실제 API 응답 구조와 반드시 동일**하게 작성합니다.

```ts
// ✅ 좋은 예 — 실제 /api/builder/pages 응답 구조 반영
await route.fulfill({
    body: JSON.stringify({ ok: true, pages: MOCK_PAGES_LIST }),
});

// ❌ 나쁜 예 — 실제 API는 data가 아닌 pages를 반환
await route.fulfill({
    body: JSON.stringify({ ok: true, data: MOCK_PAGES_LIST }),
});
```

**API 응답 구조 변경 시 관련 mock 데이터를 반드시 동기화합니다.** mock과 실제 API가 불일치하면 테스트가 자기 자신만 검증하게 됩니다.

### page.route() 사용 패턴

`page.route()`는 브라우저 내부 요청만 인터셉트합니다. `page.request`(APIRequestContext)에는 적용되지 않습니다.

```ts
// ✅ mock이 필요한 경우 — page.evaluate() 안에서 fetch 호출
await page.route('**/api/components', async (route) => {
    await route.fulfill({ body: JSON.stringify({ ok: true, components: [...] }) });
});
await page.goto('/');  // origin 확보 (about:blank 사용 금지 — 상대 URL 불가)
const result = await page.evaluate(async () => {
    const res = await fetch('/api/components');
    return res.json();
});

// ❌ 금지 — page.request는 page.route() mock을 무시하고 실제 API 호출
const res = await page.request.get('/api/components');
```

### test.skip 사용 기준

CI 환경의 **구조적 한계**(Oracle 없음, DB 의존 UI 등)일 때만 허용합니다.

```ts
// ✅ 좋은 예 — 구체적인 이유 명시
// eslint-disable-next-line playwright/no-skipped-test
test.skip(!!process.env.CI, 'CI 환경: Oracle 없어 ContentBuilder 초기화 불가 — 로컬에서만 실행');

// ❌ 나쁜 예 — 이유 없는 skip, 또는 타임아웃을 늘리는 임시방편 후 skip
test.skip(true, '나중에 고치기');
```

### UI 텍스트 관리

버튼·라벨 등 셀렉터에 쓰이는 텍스트는 `e2e/fixtures/locale.ts`의 `LABEL` 상수로 중앙 관리합니다. 텍스트가 변경되면 이 파일만 수정합니다.

```ts
// ✅
import { LABEL } from '../fixtures/locale';
page.getByRole('button', { name: LABEL.editor.save });

// ❌
page.getByRole('button', { name: 'Save' }); // 문자열 직접 사용
```
