# 전체 워크플로 분석 — 최초 페이지 제작 ~ 대고객 배포

> 최종 갱신: 2026-04-23

---

## 시스템 구성 (3자 구조)

| 시스템 | 기술 스택 | 역할 |
|---|---|---|
| Spider Admin | Spring Boot (별도 프로젝트) | 관리자 기능 — 로그인/권한/역할, 승인 UI, 작업 이력 로그 |
| CMS :3000 | Node.js (동일 소스) | 현업 관리자용 CMS — 이미지/템플릿 제작, 결재 전 리소스 저장 |
| CMS :3001 | Node.js (동일 소스) | 현업 제작자용 CMS + 대고객 서빙 — 페이지 제작, 결재 후 리소스 저장 |

### 인프라 구조

```
Spider Admin (Spring Boot) — 별도 프로젝트
├── 로그인/권한/역할 관리
├── CMS 관리자 기능 이식 (승인 목록, 승인/반려 UI)
├── 작업 이력 로그
└── CMS API 호출하여 승인/배포 처리

CMS :3000 — 현업 관리자용
├── 콘텐츠 제작 관련 기능 (에디터, 대시보드)
├── 관리자 기능은 없음 (Spider Admin에서 처리)
├── 볼륨: /data/uploads — 결재 전 이미지·리소스
└── 사용자: 현업 관리자

CMS :3001 — 제작자 + 대고객용
├── 현업 제작자용 CMS (에디터, 대시보드)
├── 대고객 콘텐츠 배포/서빙
├── 볼륨: /data/deployed — 배포된 페이지 HTML + /data/deployed/img/ (승인된 이미지)
├── 볼륨: /data/uploads (읽기 전용) — 에디터에서 결재 전 이미지 접근용
└── 사용자: 현업 제작자 + 일반 사용자
```

- CMS 두 서버는 동일 소스(Docker 이미지)로 배포
- DB는 공유 Oracle (양쪽 서버에서 동일 데이터 접근)
- 페이지·템플릿 HTML은 DB CLOB에 저장 (`SPW_CMS_PAGE.PAGE_HTML`)
- 이미지는 DB에 바이너리 미저장, 서버 내 물리 파일로 저장
- **저장소 물리 분리**: `/data/uploads/`(결재 전) ↔ `/data/deployed/`(결재 후)
- 승인 시 이미지 파일을 `/data/uploads/` → `/data/deployed/img/`로 복사 (페이지 HTML과 분리)
- 배포 시 페이지 HTML 문서를 `/data/deployed/{pageId}.html`에 생성 (`deploy/push` → `deploy/receive`)
- `/data/deployed/` 하위 구조: `img/`(승인된 이미지) + `{pageId}.html`(배포 페이지)
- :3001은 `/data/deployed/` 마운트 + `/data/uploads/` 읽기 전용 마운트 → 에디터에서 이미지 접근 가능
- 미승인 이미지 업로드는 :3000에서만 수행 (읽기 전용으로 쓰기 차단)
- 이미지 경로는 서브도메인 매핑으로 물리 경로 노출 방지 (`img.mydomain.com`)
- nginx 리버스 프록시를 통해 외부 접근

## 전체 워크플로

### 0단계: 이미지 준비

```
현업 관리자 (:3000)
  이미지 업로드
    → :3000 볼륨 /data/uploads/에 물리 파일 저장 + DB 메타데이터 등록 (ASSET_STATE = 미승인)
    → Spider Admin에서 결재자가 승인
      → DB 상태값 변경 (ASSET_STATE = 승인)
      → 이미지 파일을 /data/uploads/ → /data/deployed/img/로 복사
        → :3001 에디터에서 승인된 이미지 사용 가능
```

- 이미지가 먼저 승인되어야 이후 모든 단계에서 사용 가능
- 저장소 물리 분리: 미승인 이미지는 :3000(`/data/uploads/`)에만 존재, 승인 후 `/data/deployed/img/`로 복사
- :3001은 `/data/uploads/` 읽기 전용 마운트 → 에디터에서 이미지 조회 가능, 업로드는 불가
- DB에는 물리 경로, 서브도메인 매핑 주소, 승인 상태 관리 (바이너리 미저장)

### 1단계: 템플릿 제작 (현업 관리자)

```
현업 관리자 (:3000)
  빈 화면 + 승인된 이미지로 부서별 템플릿 제작
  (카드팀 템플릿, 뱅킹팀 템플릿 등)
    → DB에 HTML 저장 (공유 Oracle, 상태 = 미승인)
    → Spider Admin에서 결재자가 승인
      → DB 상태값 변경 → 승인된 템플릿이 :3001에서 선택 가능 상태
```

- 빈 화면에서 시작할 수 있는 권한은 현업 관리자(:3000)만 보유
- 만든 템플릿이 현업 제작자용 카테고리별 템플릿이 됨
- 템플릿 HTML은 DB에 저장, 승인 상태로 :3001 노출 제어

### 2단계: 페이지 제작 (현업 제작자)

```
현업 제작자 (:3001)
  자기 부서 카테고리의 승인된 템플릿 선택
  (빈 화면 시작 불가)
    → 내용 채우기 (텍스트 수정, 승인된 이미지만 사용)
    → DB에 저장 (공유 DB)
    → Spider Admin에서 결재자가 승인
```

### 3단계: 대고객 배포

```
Spider Admin에서 결재자가 승인 완료
  → deploy/push API 호출
    → DB에서 페이지 HTML 조립 (CSS 인라인 + 이미지 경로 치환 + 트래커 삽입)
    → 완성된 HTML 문서를 :3001 /data/deployed/{pageId}.html에 저장
    → 참조 이미지는 승인 시 이미 /data/deployed/img/에 복사되어 있음
  → 일반 사용자가 :3001에서 열람
  → HTML 내 이미지 경로는 서브도메인(img.mydomain.com)으로 노출
```

### 전체 그림

```
[0단계 이미지]   관리자(:3000) 업로드 → /data/uploads/ 저장 → Spider 승인 → /data/deployed/img/로 복사
                                                                            ↓ 승인된 이미지 사용
[1단계 템플릿]   관리자(:3000) 제작   → DB 저장              → Spider 승인 → DB 상태값 변경
                                                                            ↓ 승인된 템플릿 선택
[2단계 페이지]   제작자(:3001) 제작   → DB 저장              → Spider 승인 → /data/deployed/ 배포
                                                                            ↓
[3단계 서빙]     일반 사용자(:3001)가 열람
```

## 이미지 관리 정책

| 항목 | 정책 |
|---|---|
| 저장 방식 | 서버 내 물리 파일 (DB에 바이너리 미저장) |
| 저장소 분리 | 미승인: `/data/uploads/`(:3000) → 승인 후: `/data/deployed/img/`(:3001)로 복사 |
| DB 저장 내용 | 물리 경로, 서브도메인 매핑 주소, 승인 상태(`ASSET_STATE`) |
| 승인 제어 | DB 상태값 + 물리적 파일 분리 (이중 방어) |
| 경로 노출 방지 | 서브도메인 매핑 (`img.mydomain.com → 이미지 서버`) |
| 사용 제한 | 승인된 이미지만 에디터에서 사용 가능 |
| 승인 주체 | 결재자 (Spider Admin에서 CMS API 호출) |
| 업로드 주체 | 현업 관리자 (:3000) |

## 승인 처리 구조

승인 UI는 Spider Admin에 있고, 실제 처리는 CMS API를 호출하는 구조:

```
Spider Admin (승인 UI)
  → POST /cms/api/builder/pages/{id}/approve-request  ← CMS API (승인 요청)
  → POST /cms/api/builder/pages/{id}/approve          ← CMS API (승인 처리 + 이미지 파일 복사)
  → POST /cms/api/builder/pages/{id}/reject           ← CMS API (반려)
  → POST /cms/api/deploy/push                         ← CMS API (배포 트리거)
```

- 승인 비즈니스 로직(DB UPDATE, 파일 복사, 배포)은 CMS에 이미 구현되어 있음
- Spider Admin은 UI만 제공하고, CMS API를 호출하여 처리
- CMS 자체의 관리자 화면(승인 목록 등)은 Spider Admin으로 이식 예정
